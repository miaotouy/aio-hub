# 🔧 上下文管道

上下文管道 (Context Pipeline) 是 LLM Chat 的核心处理引擎——它负责将原始消息、预设、世界书、知识库等内容**组装成最终发送给 LLM 的请求上下文**。

## 为什么需要管道？

一次 AI 对话请求涉及多个处理步骤：

1. 加载会话历史消息
2. 应用文本替换规则
3. 注入 Agent 预设消息（系统提示、示例等）
4. 匹配并注入世界书条目
5. 检索知识库相关内容
6. 处理附件转写结果
7. 展开宏和变量
8. 限制 Token 预算
9. 格式化最终输出

上下文管道以**可编排的处理器序列**来管理这些步骤，每个处理器专注于一个职责。

## 管道架构

```
原始数据 → [处理器1] → [处理器2] → [处理器3] → ... → 最终上下文
```

### 默认处理器执行顺序

| 处理器                      | 职责                                                          |
| --------------------------- | ------------------------------------------------------------- |
| **Session Loader**          | 从树状会话中提取当前活跃路径的线性消息列表                    |
| **Regex Processor**         | 执行 Global/Agent/User 三层正则替换规则                       |
| **Injection Assembler**     | 处理复杂的预设消息注入（含深度注入和锚点定位）                |
| **Knowledge Processor**     | 执行 RAG 检索，将知识库条目注入上下文                         |
| **Transcription Processor** | 处理附件的转写结果及纯文本文件的内容提取                      |
| **Worldbook Processor**     | 扫描关键词并注入关联的世界书条目                              |
| **Token Limiter**           | 核心预算管理，按优先级裁剪超长上下文                          |
| **Async Task Processor**    | 汇总并注入关联的异步任务执行状态                              |
| **Variable Processor**      | 处理会话变量的动态展开与状态更新                              |
| **Message Formatter**       | 调整消息结构以符合特定厂商（OpenAI/Claude/Gemini）的 API 要求 |
| **Asset Resolver**          | 最后一步：将资产占位符转换为最终的 Base64 或资源 URL          |

## 管道配置

在聊天设置中打开「上下文管道」折叠区域：

- **启用/禁用处理器**：每个处理器都可独立开关
- **调整顺序**：拖拽改变处理器的执行顺序
- **恢复默认**：一键重置为出厂配置

### 何时需要调整管道？

- **不需要世界书**：关闭 Worldbook Processor 节省 Token
- **不需要知识库**：关闭 Knowledge Processor
- **自定义处理顺序**：高级用户按需调整

## Token 预算管理

Token Limiter 处理器负责确保上下文不超过模型的 Token 上限。

### 裁剪策略

当上下文超长时，系统会按以下优先级裁剪：

1. 最早的历史消息（最重要）
2. 世界书条目（次重要）
3. 预设消息（通常保留）

### 上下文压缩

你也可以手动或自动压缩历史消息——将一段较长的对话摘要为简短的总结，大幅减少 Token 占用。压缩后的消息节点会标记为压缩节点，原始消息被隐藏。
