# 知识库架构优化 - 重构总结

## 问题诊断

原 commit `8dc557be` 存在**过度拆分**问题：

### 1. 文件碎片化严重
- 将 `kbIndexer.ts` (490行) 拆成 4 个极小文件（40-90行）
- 每个文件只有 1-2 个函数，导入复杂度激增

### 2. 编排器层级冗余
- 创建了 3 个独立编排器类（IndexingOrchestrator、SearchOrchestrator、VectorSyncManager）
- 存在循环依赖和职责重叠

### 3. 适配器层过度抽象
- `BackendAdapter` 只是简单封装 Tauri invoke，无实际价值
- 增加了一层间接调用，降低可读性

### 4. 违背架构原则
- 把纯函数逻辑包装成类和实例方法
- 引入了不必要的状态管理

## 重构方案

采用**"适度聚合 + 清晰分层"**策略：

### 新架构（三层）

```
core/                          # 纯函数层（无状态）
├── embedding.ts               # 向量生成、标签向量化、维度探测（210行）
├── search.ts                  # 搜索向量准备（60行）
└── tagGenerator.ts            # 标签生成逻辑（内联 retry）

logic/                         # 业务编排层
└── orchestrator.ts            # 统一编排器（400行）
    ├── IndexingOrchestrator   # 索引编排
    ├── VectorSyncManager      # 向量同步
    └── SearchOrchestrator     # 检索编排

composables/                   # UI 交互层
├── useKbIndexer.ts           # 索引管理
├── useKbVectorSync.ts        # 同步管理
└── useKnowledgeSearchManager.ts  # 搜索管理
```

### 核心改进

#### 1. Core 层合并（纯函数集合）
**之前**：
- `core/embedding/vector-generator.ts` (54行)
- `core/embedding/tag-vectorizer.ts` (92行)
- `core/search/vector-search-prepper.ts` (47行)
- `core/utils/retry.ts` (41行)

**之后**：
- `core/embedding.ts` (210行) - 包含所有向量相关纯函数
- `core/search.ts` (60行) - 包含搜索相关纯函数

**优势**：
- 相关逻辑聚合，减少文件跳转
- 保持纯函数特性，易于测试
- 内部 retry 逻辑复用，无需单独文件

#### 2. Logic 层统一（单文件编排）
**之前**：
- `logic/adapters/BackendAdapter.ts` (104行)
- `logic/orchestrators/IndexingOrchestrator.ts` (190行)
- `logic/orchestrators/SearchOrchestrator.ts` (143行)
- `logic/orchestrators/VectorSyncManager.ts` (78行)

**之后**：
- `logic/orchestrator.ts` (400行) - 包含三个编排器类

**优势**：
- 去除冗余适配器层，直接使用 Tauri invoke
- 编排器职责清晰，相互协作更直观
- 减少循环依赖风险

#### 3. Composables 层简化
**改进**：
- 直接导入统一编排器，无需多层实例化
- 去除 BackendAdapter 中间层
- 类型安全，导入路径简洁

## 架构原则遵循

✅ **逻辑函数式化**：Core 层保持纯函数，无状态  
✅ **适度聚合**：相关功能聚合在一起，避免过度拆分  
✅ **清晰分层**：Core → Logic → Composables 职责明确  
✅ **去除冗余**：删除无实际价值的适配器层  
✅ **直接调用**：Composables 直接使用 Tauri invoke，减少间接层

- ✅ TypeScript 类型检查通过

## 经验总结

### ❌ 错误做法
1. **一个函数一个文件** - 导致文件爆炸
2. **过度抽象** - 简单封装 invoke 的适配器层
3. **类包装纯函数** - 违背函数式原则
4. **多层编排器** - 职责重叠，循环依赖

### ✅ 正确做法
1. **按功能模块聚合** - 相关函数放在一起
2. **直接使用底层 API** - 避免无意义的封装
3. **保持纯函数** - Core 层无状态，易测试
4. **统一编排层** - 单文件包含所有编排逻辑

## 结论

本次重构成功解决了过度拆分问题，在保持代码清晰度的同时：
- 减少了 4 个文件
- 降低了导入复杂度
- 提升了代码可维护性
- 符合项目架构规范

重构后的架构更加**简洁、直观、易维护**，真正实现了"解耦"而非"拆文件"。
