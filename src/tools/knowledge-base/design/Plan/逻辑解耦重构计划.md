# 知识库逻辑解耦重构计划

## 问题概述

当前知识库工具的核心业务逻辑与 UI 层（Composables、Store）严重耦合，导致：

1. 代码难以测试（需要模拟整个 Vue 环境）
2. 逻辑难以复用（移动端、CLI 工具等场景无法直接使用）
3. 职责不清晰（Composables 既管理状态又执行业务逻辑）
4. 违反了项目规范中的"逻辑函数式化"原则

## 重构目标

遵循 **"Functional Core, Imperative Shell"** 架构模式：

- **Core (核心层)**: 纯函数逻辑，无副作用，不依赖 Vue
- **Logic (逻辑层)**: 协调多个 Core 函数，处理复杂流程
- **Composables (粘合层)**: 仅作为 UI 与 Logic 之间的适配器

### 阶段三：重构 Composables 为粘合层（优先级：中）

#### 3.1 简化 `useKbIndexer`

**重构后职责**:

- 管理 UI 状态（loading、progress）
- 调用 `IndexingOrchestrator`
- 处理用户交互（消息提示、错误处理）

**示例**:

```typescript
// composables/useKbIndexer.ts
export function useKbIndexer() {
  const store = useKnowledgeBaseStore();
  const { profiles } = useLlmProfiles();

  // 创建编排器实例（可以缓存）
  const orchestrator = new IndexingOrchestrator(backendAdapter, vectorGenerator, store.config);

  const loading = ref(false);
  const progress = ref({ current: 0, total: 0 });

  async function indexEntry(entryId: string) {
    loading.value = true;
    try {
      const entry = await store.getOrLoadEntry(entryId);
      const profile = getProfile(store.config.defaultEmbeddingModel);

      // 调用逻辑层
      const result = await orchestrator.indexEntry({
        kbId: store.activeBaseId!,
        entry,
        modelId: getPureModelId(store.config.defaultEmbeddingModel),
        profile,
      });

      // 更新 UI 状态
      store.vectorizedIds.add(entryId);
      customMessage.success("索引完成");
    } catch (error) {
      errorHandler.error(error, "索引失败");
    } finally {
      loading.value = false;
    }
  }

  return { loading, progress, indexEntry };
}
```

## 重构收益

### 1. 可测试性

- Core 层纯函数可以直接单元测试
- Logic 层可以通过 Mock BackendAdapter 测试
- Composables 变薄，测试负担减轻

### 2. 可复用性

- Core 和 Logic 层可以在移动端直接复用
- 可以轻松创建 CLI 工具或 Node.js 脚本
- 便于构建独立的测试工具

### 3. 可维护性

- 职责清晰，每层只做一件事
- 依赖关系单向（Core ← Logic ← Composables）
- 易于定位问题和添加新功能

### 4. 符合规范

- 遵循移动端规范的"逻辑函数式化"原则
- 符合项目整体架构设计
- 为未来的跨平台扩展打下基础
