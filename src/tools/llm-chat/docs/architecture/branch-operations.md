## 1. 核心概念

### 1.1 树结构

会话的消息历史是一棵**多叉树**，而非线性列表：

- **`ChatMessageNode`**：消息节点，通过 `parentId` → `childrenIds[]` 建立父子关系
- **`rootNodeId`**：每个会话有一个根节点（`role: "system"`，内容为空），作为树的起点
- **`activeLeafId`**：当前活跃叶节点 ID，决定 UI 显示哪条对话路径
- **`nodes`**：以 `nodeId` 为 key 的字典，存储所有节点

### 1.2 分支（Branch）

分支指从同一个父节点出发的不同子路径。例如，对同一用户消息重新生成两次，就会产生两个不同的助手消息分支。

### 1.3 分支记忆（Branch Memory）

每个节点可以通过 `lastSelectedChildId` 记住用户上次选择的是哪个子节点。当从其他分支切回时，系统可以自动导航到上次查看的位置。

### 1.4 撤销/重做栈

采用**快照 + 增量**混合策略：

- **快照（Snapshot）**：定期或复杂操作后记录完整的 `nodes` 状态
- **增量（Delta）**：轻量级操作只记录变更差异（`create` / `delete` / `update` / `relation` / `active_leaf_change`）

历史记录在**发送新消息**或**重新生成**时被清空（历史断点）。

## 2. 操作分类总览

| 类别          | 操作           | 用户触发方式                  | 对树的影响                          |
| ------------- | -------------- | ----------------------------- | ----------------------------------- |
| **树结构**    | 创建分支       | 消息菜单 → 创建分支按钮       | 新增兄弟节点，切换 activeLeaf       |
|               | 切换分支       | 分支选择器 / Prev/Next        | 更新 activeLeaf，更新分支记忆       |
|               | 编辑消息       | 消息菜单 → 编辑               | 原地修改节点内容（不改变树结构）    |
|               | 从编辑创建分支 | 编辑对话框 → 保存到分支       | 新增兄弟节点，切换 activeLeaf       |
|               | 删除消息       | 消息菜单 → 删除（需二次确认） | 硬删除节点及其子树，调整 activeLeaf |
|               | 切换启用状态   | 消息菜单 → 禁用/启用          | 翻转 `isEnabled` 布尔值             |
| **消息生成**  | 发送消息       | 输入框 → 发送                 | 创建用户+助手消息对，清空历史       |
|               | 重新生成       | 消息菜单 → 重新生成           | 创建新的助手回复分支，清空历史      |
|               | 续写消息       | 更多菜单 → 续写               | 创建续写节点，清空历史              |
| **可视化图**  | 嫁接分支       | 树图拖拽                      | 节点子树移动到新父节点              |
|               | 移动节点       | 树图拖拽（带 modifier）       | 节点本身移动，子节点被收养          |
| **撤销/重做** | 撤销/重做      | 快捷键 / UI 按钮              | 恢复到历史快照并重放增量            |

## 3. 树结构操作

### 3.1 创建分支（Create Branch）

**触发方式**：用户点击消息菜单栏的「创建分支」按钮（`GitFork` 图标），在任意用户/助手消息上均可触发。

**生效逻辑**：

1. **验证源节点**：
   - 源节点必须在 `session.nodes` 中存在
   - 允许 `role === "user"` 或 `role === "assistant"` 的节点
   - 预设消息（`metadata.isPresetDisplay === true`）不支持创建分支

2. **创建新节点**：
   - 新节点的 `parentId` 指向源节点的 `parentId`（即成为源节点的兄弟）
   - `role`、`content` 全部复制自源节点
   - `attachments` 如果存在则深拷贝（使用展开运算符创建新数组）
   - 初始 `status: "complete"`，`isEnabled: true`

3. **元数据处理（关键差异）**：
   - **助手消息**：使用浅拷贝复制源节点的 `metadata`（`{ ...sourceNode.metadata }`）
   - **用户消息**：
     - 如果源节点已有 `userProfileId`（历史快照），使用浅拷贝复制整个 `metadata`，保持历史一致性
     - 如果源节点没有（旧消息），**捕获当前生效的用户档案**作为新节点的身份快照：
       - **优先级 1**：当前 Agent 绑定的用户档案（`agent.userProfileId`）
       - **优先级 2**：全局默认用户档案（`userProfileStore.globalProfile`）
       - 保存字段：`userProfileId`、`userProfileName`（优先 `displayName`，回退 `name`）、`userProfileIcon`

4. **添加到会话**：
   - 将新节点注册到 `session.nodes`
   - 更新父节点的 `childrenIds`（如果父节点存在且新节点 ID 不在列表中，则追加）

5. **切换 activeLeaf**：`session.activeLeafId` 指向新节点

6. **更新分支记忆**：调用 `BranchNavigator.updateSelectionMemory`，从新叶节点到根路径上所有父节点的 `lastSelectedChildId` 被更新

7. **记录历史**：生成 `type: "create"` 的 `HistoryDelta`（包含新节点快照和关系变化）

8. **持久化**：更新消息计数、更新会话显示智能体、持久化到文件

### 3.2 切换分支（Switch Branch）

**触发方式**：

- 在分支选择器 `BranchSelector` 中点击某个分支项
- 点击消息菜单栏的 Prev/Next 箭头切换兄弟分支
- 在树图视图中点击某个节点

#### 3.2.1 直接切换（switchBranch）

**生效逻辑**：

1. 定位目标节点，验证存在性
2. 将 `session.activeLeafId` 设置为目标节点 ID
3. **更新分支记忆**：从目标叶节点到根路径上所有父节点逐层更新 `lastSelectedChildId`，确保路径上的每个父节点都记住用户这次的选择
4. 如果 activeLeaf 实际发生了变化，记录 `type: "active_leaf_change"` 的 `HistoryDelta`
5. 更新 UI 显示（消息列表重新渲染）
6. 更新会话显示智能体、持久化

**注意**：切换分支只是修改了 `activeLeafId` 指针，树结构本身没有任何变化。

#### 3.2.2 兄弟切换（switchToSiblingBranch）

**生效逻辑**：

1. 获取当前节点所在父节点的所有子节点（兄弟列表）
2. 根据方向（`prev` / `next`）计算目标兄弟索引（支持循环切换）
3. 从目标兄弟节点调用 `findLeafOfBranch` 找到其分支的叶节点：
   - **优先**使用目标兄弟节点的 `lastSelectedChildId`（分支记忆）
   - **回退**使用第一个子节点
   - 递归直到找到叶子
4. 将 `session.activeLeafId` 设置为找到的叶节点
5. 更新分支记忆
6. 持久化

**分支记忆导航流程图**：

```
从节点 A 切换到兄弟 B：
1. 找到 B 节点
2. B 有子节点吗？
   ├── B.lastSelectedChildId 存在 → 沿着记忆路径走到叶子
   └── 没有记忆 → 沿着第一个子节点走到叶子
3. 更新 activeLeafId = 叶节点 ID
4. 更新路径上所有父节点的 lastSelectedChildId
```

### 3.3 编辑消息（Edit Message）

**触发方式**：用户点击消息菜单的「编辑」按钮，修改内容后保存。

**生效逻辑**（有两种模式）：

#### 模式 A：原地编辑（直接保存）

1. 验证节点角色（只允许 `user` 和 `assistant`）
2. **直接修改** `node.content` 为新的内容
3. **选择性更新**附件（如果提供了则替换，空数组则删除附件字段）
4. 更新时间戳
5. 记录 `type: "update"` 的 `HistoryDelta`，包含修改前和修改后的节点快照
6. 重新计算 Token
7. 持久化

**特点**：不改变树结构，不创建新节点，操作可撤销。

#### 模式 B：从编辑创建分支（save to branch）

**触发方式**：编辑对话框中的「保存到分支」选项。

**生效逻辑**：

1. 创建新节点，`role` 和 `parentId` 与源节点相同（即成为兄弟节点）
2. 新节点的 `content` 设置为编辑后的内容
3. 元数据清理：剥离所有执行相关字段（`error`、`usage`、`tokenCount`、`reasoningContent` 等），保留用户档案等身份信息
4. 添加到会话（更新父子关系）
5. **切换 activeLeaf** 到新节点
6. 更新分支记忆
7. 记录历史、重新计算 Token、持久化

**本质**：相当于 `createBranch` + 内容替换的组合操作。

### 3.4 删除消息（Delete Message / Hard Delete）

**触发方式**：用户点击消息菜单的「删除」按钮，在二次确认弹窗中确认后执行。

> **注意**：这里的删除是**硬删除**（从树中物理移除），非软删除（禁用节点）。

**生效逻辑**：

1. **前置验证**：
   - 节点必须存在于 `session.nodes`
   - 不能删除根节点（`node.id === session.rootNodeId`）

2. **压缩节点特殊处理**（如果删除的是压缩节点）：
   - 不递归删除子节点，而是将子节点列表「归还」给父节点
   - 更新被归还子节点的 `parentId`
   - 从父节点的 `childrenIds` 中用子节点列表替换压缩节点

3. **普通节点**（非压缩节点）：
   - **递归收集**所有需要删除的节点（自目标节点起的所有后代）
   - 使用 `nodesToDeleteIds` Set 收集所有受影响节点 ID

4. **activeLeaf 补偿策略**（如果当前活跃叶节点被删除）：
   - 获取被删除节点在兄弟列表中的索引
   - **优先**选择下一个兄弟的 deepest leaf
   - **其次**选择上一个兄弟的 deepest leaf
   - **兜底**回退到父节点或根节点
   - 更新分支记忆

5. **清理父子关系**：从父节点的 `childrenIds` 中移除被删除节点

6. **物理删除**：从 `session.nodes` 中 `delete` 所有被收集的节点

7. **记录历史**：生成 `type: "delete"` 的 `HistoryDelta`，每个被删除节点都包含其完整快照和关系变化信息

8. **持久化**：更新计数、显示智能体、持久化

**activeLeaf 补偿策略流程图**：

```
删除节点 N：
1. 当前 activeLeaf 是否在删除范围内？
   ├── 否 → activeLeaf 不变，直接删除
   └── 是 → 执行补偿策略
        ├── 存在下一个兄弟？→ 找到该兄弟分支的最深叶子
        ├── 存在上一个兄弟？→ 找到该兄弟分支的最深叶子
        └── 没有兄弟？→ 回退到父节点/根节点
2. 更新分支记忆
3. 执行物理删除
```

### 3.5 切换节点启用状态（Toggle Node Enabled）

**触发方式**：用户点击消息菜单的「禁用/启用」按钮（`Eye`/`EyeOff` 图标）。

**生效逻辑**：

1. 翻转节点的 `isEnabled` 布尔值（默认 `true`）
2. 记录 `type: "update"` 的 `HistoryDelta`
3. 持久化

**影响**：

- `isEnabled === false` 的节点在构建 LLM 上下文时会被 `filter` 跳过
- 不影响树结构，节点及其子节点依然存在
- 在 UI 上会被淡化显示
- 在 `getNodePath` 等路径获取方法中，禁用的节点也会被跳过

## 4. 消息生成操作

### 4.1 发送消息（Send Message）

**触发方式**：用户在输入框输入内容，点击发送按钮或按快捷键。

这是最复杂的操作，涉及多个阶段：

#### 阶段 1：前置处理

1. **上下文压缩检查**：尝试自动压缩上下文（如果达到阈值），压缩失败不阻断流程
2. **宏处理**：如果未禁用宏解析，对用户输入进行宏替换（`PRE_PROCESS` → `SUBSTITUTE` → `POST_PROCESS`）

#### 阶段 2：创建消息对

1. 使用 `createMessagePair` 创建两个节点：
   - **用户节点**：`role: "user"`，`parentId` 为当前 `activeLeafId`
   - **助手节点**：`role: "assistant"`，`parentId` 为用户节点的 ID，初始 `status: "generating"`
2. 建立双向父子关系：用户节点的 `childrenIds` 包含助手节点 ID
3. 两个节点依次添加到 `session.nodes`
4. 更新 `activeLeafId` 为助手节点 ID

#### 阶段 3：元数据填充

1. **用户节点**：保存用户档案快照（`userProfileId`、`userProfileName`、`userProfileIcon`）
2. **助手节点**：保存 Agent 和模型的配置快照（`agentId`、`agentName`、`profileId`、`modelId` 等）

#### 阶段 4：附件处理

1. 如果有附件，调用 `processUserAttachments` 处理
2. **转写等待**（如果启用且附件存在）：
   - 创建 AbortController 加入 `abortControllers`
   - 等待转写完成
   - 用户可取消等待（保留用户消息，删除助手节点）
   - 转写出错降级发送（无转写文本）

#### 阶段 5：上下文构建与请求

1. 计算用户消息的 Token
2. 保存用户消息（防止崩溃丢失）
3. 调用统一上下文管道（9 个处理器依次执行）
4. 发送 LLM 请求（流式/非流式）

#### 阶段 6：响应处理

1. 处理流式数据（节流更新节点内容）
2. 最终化节点状态（`complete` / `error`）
3. 异步触发上下文压缩

#### 阶段 7：收尾

1. **清空撤销栈**（历史断点）
2. 更新消息计数、显示智能体
3. 持久化会话

**发送消息流程图**：

```
用户点击发送
    │
    ▼
上下文压缩检查（可选的，失败不阻断）
    │
    ▼
宏处理（替换 {{user}}、{{time}} 等）
    │
    ▼
创建消息对（用户节点 + 助手节点）
    │
    ▼
填充元数据快照（Agent/模型/用户档案信息）
    │
    ▼
处理附件 + 转写等待
    │
    ▼
构建上下文（统一管道 9 步骤）
    │
    ▼
执行 LLM 请求（流式/非流式）
    │
    ▼
响应处理（流式更新、状态终态化）
    │
    ▼
清空撤销栈 ← 历史断点
    │
    ▼
持久化会话
```

### 4.2 重新生成（Regenerate）

**触发方式**：用户点击消息菜单的「重新生成」按钮，可选切换模型后重新生成。

#### 4.2.1 从指定节点重新生成（regenerateFromNode）

**生效逻辑**：

1. **定位目标节点**：
   - **用户消息**：以此用户消息为基础，创建新的助手回复作为其子节点
   - **助手消息**：取其父节点（用户消息），在该用户消息下创建新的助手回复作为兄弟

2. **创建新助手节点**：
   - `role: "assistant"`，`content: ""`，`status: "generating"`
   - 添加到会话，建立父子关系

3. **元数据填充**（同发送消息）

4. **模型切换支持**：如果用户选择了临时模型，覆盖 `agentConfig` 中的 `modelId` 和 `profileId`，并过滤只保留目标模型支持的参数

5. **执行 LLM 请求**（同发送消息的请求流程）

6. **清空撤销栈**（历史断点）

#### 4.2.2 重新生成最后一条消息（regenerateLastMessage，向后兼容）

**生效逻辑**：

1. 检查当前 `activeLeaf` 是否是助手消息
2. 如果是，找到其父节点（用户消息）
3. 将 `activeLeafId` 回退到用户消息
4. 调用 `sendMessage`，使用用户消息的内容作为输入
5. 这会导致创建一个新的用户消息 → 新的助手消息 → 形成分支

### 4.3 续写消息（Continue）

**触发方式**：用户点击更多菜单的「续写消息」或「选择模型续写」。

**生效逻辑**（有两种情况）：

#### 情况 A：Assistant 续写

1. 目标节点是助手消息
2. 创建新节点：`role: "assistant"`，`content` **复制**原助手消息的内容
3. `parentId` 指向原节点的 `parentId`（即成为兄弟节点）
4. 元数据附加 `isContinuation: true` 和 `continuationPrefix`（原始内容，用于后续拼接校验）
5. 在请求时使用 `prefix`/`prefill` 特性让模型从现有文本末尾继续生成

#### 情况 B：User 续写

1. 目标节点是用户消息
2. 创建新节点：`role: "assistant"`，`content: ""`（空，因为是角色接力）
3. `parentId` 指向该用户消息（即作为子节点）
4. 模型从用户视角继续书写

#### 共同步骤

1. 加入 `generatingNodes` 集合
2. 更新 `activeLeafId` 到新节点
3. 填充 Agent / 模型元数据
4. 执行 LLM 请求（带 `isContinuation` 标记）
5. 清空撤销栈 → 持久化

## 5. 可视化图操作

这些操作通过 `useGraphActions` 暴露给树图视图，支持拖拽交互。

### 5.1 嫁接分支（Graft Branch）

**触发方式**：用户在树图视图中拖拽一个节点到另一个节点上（无 modifier 键）。

此操作在 `useNodeManager.reparentSubtree` 中实现，受 `useBranchManager.graftBranch` 封装。

**生效逻辑**：

1. **验证操作合法性**：
   - 源节点、目标父节点必须存在
   - 不能嫁接根节点
   - 不能嫁接到自己
   - **防止循环引用**：检查新父节点不能是源节点的子孙（否则会形成环）

2. **如果已经是目标父节点的子节点**：直接返回成功（空操作）

3. **从旧父节点断开**：
   - 从旧父节点的 `childrenIds` 中移除源节点 ID
   - 记录旧关系变化（用于撤销）

4. **挂接到新父节点**：
   - 更新源节点的 `parentId` 为目标父节点 ID
   - 将源节点添加到新父节点的 `childrenIds`

5. **记录历史**：生成 `type: "relation"` 的 `HistoryDelta`，包含所有关系变化

6. **持久化**：更新计数、显示智能体、持久化

**安全保护**：

- 循环引用检测：递归收集源节点的所有子孙节点，检查新父节点是否在其中
- 如果检测到循环，操作失败并记录警告

**嫁接流程图**：

```
用户将节点 N 拖拽到新父节点 P
    │
    ▼
检查：N 存在吗？P 存在吗？→ 否 → 失败
    │
    ▼
检查：N 是根节点吗？→ 是 → 失败
    │
    ▼
检查：N === P？→ 是 → 失败
    │
    ▼
检查：P 是 N 的子孙吗？→ 是 → 失败（循环引用）
    │
    ▼
从旧父节点的 childrenIds 中移除 N
    │
    ▼
更新 N.parentId = P.id，添加 N 到 P.childrenIds
    │
    ▼
记录关系变更 delta → 持久化
```

### 5.2 移动节点（Move Node）

**触发方式**：用户在树图视图中拖拽一个节点到另一个节点上（带 modifier 键，如 Ctrl/Shift）。

此操作在 `useNodeManager.reparentNode` 中实现，受 `useBranchManager.moveNode` 封装。

**生效逻辑**（与嫁接的主要区别）：

1. **验证操作合法性**（同嫁接）

2. **子节点被「收养」**（关键区别）：
   - 源节点的所有子节点**不随源节点移动**
   - 而是将源节点的 `childrenIds` 全部**追加到旧父节点**的 `childrenIds` 中
   - 更新每个子节点的 `parentId` 为旧父节点 ID
   - 清空源节点的 `childrenIds`

3. **源节点挂接到新父节点**（同嫁接）

4. **记录历史**（同嫁接）

**为什么需要子节点收养？**
移动节点通常用于将某个节点「提升」或「转移」到树的另一个位置，但保留其子节点在原来的上下文中，避免破坏原有对话路径的连续性。

**移动 vs 嫁接对比**：

| 特性       | 嫁接（Graft）    | 移动（Move）   |
| ---------- | ---------------- | -------------- |
| 子节点     | 随源节点一起移动 | 被旧父节点收养 |
| 树结构变化 | 子树整体迁移     | 单个节点迁移   |
| 典型场景   | 调整对话分支归属 | 修复节点位置   |

## 6. 副作用系统

### 6.1 撤销（Undo）与重做（Redo）

**生效逻辑**：

```
撤销：
    ├── 检查 historyIndex > 0
    ├── jumpToState(historyIndex - 1)
    │   ├── 找到最近的快照锚点（isSnapshot === true）
    │   ├── 从锚点恢复完整 nodes 快照
    │   ├── 重放从锚点到目标索引的所有增量（deltas in forward）
    │   └── 更新 historyIndex
    ├── 确保 activeLeaf 有效性（防止恢复后指向不存在的节点）
    ├── 更新消息计数
    └── 持久化

重做：
    ├── 检查 historyIndex < history.length - 1
    ├── jumpToState(historyIndex + 1)
    │   ├── 找到最近的快照锚点
    │   ├── 从锚点恢复
    │   └── 重放到目标索引
    ├── 确保 activeLeaf 有效性
    ├── 更新消息计数
    └── 持久化
```

**增量类型与处理方向**：

| 增量类型             | 正向（重做）            | 反向（撤销）            |
| -------------------- | ----------------------- | ----------------------- |
| `create`             | 恢复节点 + 恢复父子关系 | 移除父子关系 + 删除节点 |
| `delete`             | 移除父子关系 + 删除节点 | 恢复节点 + 恢复父子关系 |
| `update`             | 应用 finalNodeState     | 应用 previousNodeState  |
| `relation`           | 应用 newChildren 关系   | 应用 oldChildren 关系   |
| `active_leaf_change` | 设置 newLeafId          | 设置 oldLeafId          |

### 6.2 分支记忆更新

每次 `activeLeafId` 发生变化时（切换分支、创建分支、发送消息、重新生成等），都会调用 `updateSelectionMemory`：

1. 从新的叶节点向上遍历到根节点，收集完整路径
2. 从上到下逐层更新每个父节点的 `lastSelectedChildId`
3. 更新前会验证子节点确实存在于父节点的 `childrenIds` 中

**效果**：当用户通过分支记忆导航回某个分支时（`findLeafOfBranch`），系统会自动沿记忆路径走到用户上次查看的位置。

### 6.3 僵死节点修复

系统通过 `watch` 监听 `generatingNodes.size` 的变化。当生成节点减少时，检查当前会话中是否还有 `status: "generating"` 但不在 `generatingNodes` 集合中的节点：

- 如果有内容 → 标记为 `complete`
- 如果没有内容 → 标记为 `error`，附带错误信息 "生成意外中断"

这是在极端情况下的自愈机制（如程序崩溃、窗口被关闭等）。

### 6.4 历史断点（clearHistory）

以下操作会清空撤销栈并重新初始化：

- **发送新消息**（`sendMessage`）
- **重新生成**（`regenerateFromNode`）
- **续写**（`continueGeneration`）

初始化的历史记录为一个 `INITIAL_STATE` 快照，记录当前完整的 nodes 状态。

## 7. 分支记忆与导航

### 7.1 分支记忆机制

`lastSelectedChildId` 存储在每个节点上，记录用户上次从该父节点选择了哪个子节点。

**记忆场景**：

- 切换分支时
- 创建分支时
- 发送消息时（更新到新叶子）

**使用场景**（`findLeafOfBranch`）：

- 兄弟切换时找到目标分支的叶子
- 初始化时导航到上次查看的路径
- 树图展开时定位到活跃分支

### 7.2 活动节点有效性保障

`ensureValidActiveLeaf` 在以下场景被调用：

- 撤销/重做后
- 跳转历史状态后

如果 `activeLeafId` 指向的节点不存在（如因历史跳转导致），则重置为 `rootNodeId`。

### 7.3 预设消息的处理

分支操作对预设消息（`metadata.isPresetDisplay === true`）有限制：

- **不允许创建分支**：显示禁用提示 "预设消息暂不支持创建分支"
- **不允许重新生成**：显示禁用提示 "预设消息不参与重试"
- **不允许删除**：提示 "预设消息需要在智能体设置中编辑或删除"
- 编辑操作会直接修改 Agent 存储中的预设消息（而非会话中的节点）

## 8. 常见操作链路示例

### 链路 A：多轮对话 + 分支探索

```
1. 用户发送 "你好" → sendMessage
   → root → [用户] → [助手1]

2. 用户不满意，点击助手1的「重新生成」
   → root → [用户] → [助手1]
                      └── [助手2] (new activeLeaf)

3. 用户对助手2发问 "具体说说"
   → root → [用户] → [助手1]
                      └── [助手2] → [用户2] → [助手3]

4. 用户切回助手1查看，再切回
   → root → [用户] → [助手1]
                      ├── [助手2] → [用户2] → [助手3] ← 分支记忆
```

### 链路 B：编辑 + 分支保存

```
1. 用户编辑自己的消息，选择「保存到分支」
   → root → [用户1] → [助手1]
              └── [用户1-edited] (new activeLeaf)
                    └── (等待下一步)
```

### 链路 C：删除 + activeLeaf补偿

```
当前 activeLeaf 在 [助手2]，删除 [助手2]：

→ root → [用户] → [助手1]
                   └── [助手2] ← 被删除
                   └── [助手3] ← 下一个兄弟，activeLeaf 切到这里的最深叶子
```

### 链路 D：树图嫁接调整

```
目标：将 [助手2] 移动到 [用户1] 下（作为其子节点）

嫁接前：
→ [用户1] → [助手1] → [用户2] → [助手2]

嫁接后：
→ [用户1] → [助手1] → [用户2]
          └── [助手2] ← 连子树一起移动
```
