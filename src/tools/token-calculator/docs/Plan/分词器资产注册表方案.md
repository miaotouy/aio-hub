# 分词器资产注册表方案

## 1. 背景

当前 Token Calculator 工具依赖一组打包进 bundle 的 `@lenml/tokenizer-*` 包，并在
[`tokenCalculatorEngine.ts`](../../core/tokenCalculatorEngine.ts:1) 里硬编码了
「模型 ID → tokenizer 名称」的正则映射。

这种实现的问题：

- 模型发布速度远快于 bundle 更新；
- 一些供应商（例如较新的 Claude）不公开本地分词器，需要做近似估算；
- 用户通过第三方网关访问模型时，官方 token 计数 API 通常也不可用；
- 把所有 tokenizer 全部打进 bundle，体积会越来越大。

AIO Hub 始终坚持**本地优先 + 渠道无关**的设计，因此分词器系统应从「固定依赖列表」
演化为一个**本地资产注册表**：支持内置分词器、用户导入、远端按需下载、以及对未知
分词器的保守校准估算。

## 2. 目标

- 在没有官方 API Key 的情况下也能算 token；
- 用户可在安装后**导入或下载**新的分词器配置；
- 默认安装包保持精简；
- 保留现有的跨模块服务接口（LLM Chat 等的调用方式不变）；
- 分词器解析结果对用户透明（精确 / 近似 / 估算）；
- 对无法精确分词的模型（如新 Claude）提供**可配置的保守倍率**。

## 3. 非目标

- **不** 把官方 token 计数 API 作为主路径；
- **不** 执行下载得到的代码，下载的资产必须是纯数据；
- **不** 把分词器管理放进全局设置或单个聊天 Agent；
- **不** 要求每个 provider 或模型预设都自带分词器；
- **不** 改变 `TokenCalculationResult.count` 的现有语义（新字段只做 additive 扩展）。

## 4. 产品落点

分词器管理放在 Token Calculator 工具内部：

- 路由：`/token-calculator`
- 入口：[`token-calculator.registry.ts`](../../token-calculator.registry.ts:1)
- 主视图：[`TokenCalculator.vue`](../../TokenCalculator.vue:1)

工具内部从「单页左右分栏」改为 Workspace 风格，包含 4 个 Tab：

- `计算`：当前的文本 / 模型 token 计算器（即现有 UI）。
- `分词器库`：已安装、可下载、可导入的分词器 profile。
- `匹配规则`：模型 ID pattern → profile 映射的可视化管理。
- `校准`：multiplier、固定 overhead、buffer 等近似配置。

全局设置可后续追加一个跳转入口指向本工具，**但不持有功能本体**。

## 5. 核心架构原则

### 5.1 注册表权威态在主线程

主线程是注册表的**唯一权威态**（Single Source of Truth）：

- 注册表数据由一个独立的 Pinia store（暂命名 `useTokenizerRegistryStore`）持有；
- 持久化通过 `createConfigManager` 落到 AppData JSON（与 [`config.ts`](../../config.ts:1) 同模式）；
- 所有 UI（计算 Tab、分词器库 Tab、匹配规则 Tab、校准 Tab）都读这一份 store；
- 跨模块消费方（如 LLM Chat）通过现有 `tokenCalculatorService` 间接受益，不直接访问 store。

### 5.2 Worker 只是「无状态」镜像

Worker 内部持有的注册表数据完全由主线程通过 `init` 消息推送：

- Worker 启动 → 发 `ready` → 主线程推 `init` → Worker 切换内部状态 → 发 `initialized`；
- 安装 / 卸载 / 下载 / 改规则 / 改校准 = **低频操作** → 直接重启 Worker 重新走 init 流程；
- Worker 启动后到收到 `initialized` 之间，proxy 把请求放到队列，不直接 reject；
- 异常重启（既有逻辑）也走同一套 init 协议，所以 cold-start 路径只有一条。

### 5.3 大文件（tokenizer.json）按需推送

每个 tokenizer.json 可能几 MB，**init 阶段只推 profile 元数据**（id / name / source 标识 / 校准参数等），
真正的 JSON 内容在 Worker 第一次需要某个 profile 时按需请求：

```
Worker → 主线程: { type: "needProfileData", profileId }
主线程: 通过 Tauri 读文件
主线程 → Worker: { type: "profileData", profileId, tokenizerJSON, tokenizerConfigJSON }
Worker: 缓存 + 实例化 TokenizerLoader
```

Worker 内部对已加载的 tokenizer 做 LRU 缓存（沿用现有 `tokenizerCache` 机制）。

### 5.4 Worker 没有 Tauri API

Worker 不能调用 `@tauri-apps/plugin-fs` 或 `invoke`。所有文件 / 网络访问由主线程完成，
Worker 只接收 JSON 字符串 / ArrayBuffer。这条约束在 Phase 3 之前必须明确遵守。

## 6. 类型模型

### 6.1 Profile ID 命名空间

- **内置 profile**：ID 与现有 [`ModelMetadataProperties.tokenizer`](../../../../types/model-metadata.ts:28)
  字符串完全相同（`gpt4o` / `gpt4` / `claude` / `gemini` / `llama3_2` / `deepseek_v3` / `qwen3`），
  保证现有 metadata 规则零迁移；
- **用户导入**：ID 以 `user.` 为前缀，例如 `user.my-bert`；
- **远端下载**：ID 以 `remote.` 为前缀，例如 `remote.gptoss-o200k`。

`ModelMetadataProperties.tokenizer` 字段类型保持 `string` 不变，引擎在解析时直接拿这个
字符串当 profile ID 查注册表。

### 6.2 TypeScript 定义

```ts
export type TokenizerConfidence = "exact" | "close" | "estimated";

export type TokenizerAssetFormat =
  | "hf-tokenizer-json"
  | "hf-directory"
  | "tiktoken-bpe"
  | "sentencepiece-model"
  | "tekken-json"
  | "gguf-metadata"
  | "legacy-bpe"
  | "wordpiece-vocab"
  | "unknown";

export type TokenizerSource =
  | { type: "bundled"; packageName: string; exportName?: string }
  | {
      type: "local";
      format: TokenizerAssetFormat;
      /** HF fast / @lenml-compatible 形态才一定存在 */
      tokenizerJsonPath?: string;
      tokenizerConfigPath?: string;
      /** 慢分词器、SentencePiece、tiktoken、GGUF metadata 等其它资产 */
      assetFilePaths?: Record<string, string>;
    }
  | {
      type: "remote";
      format: TokenizerAssetFormat;
      /** HF fast / @lenml-compatible 形态才一定存在 */
      tokenizerJsonUrl?: string;
      tokenizerConfigUrl?: string;
      assetFileUrls?: Record<string, string>;
      tokenizerJsonSha256?: string;
      tokenizerConfigSha256?: string;
      assetFileSha256?: Record<string, string>;
    };

export interface TokenizerCalibration {
  /** 乘性倍率，默认 1 */
  multiplier?: number;
  /** 每次调用的固定 overhead */
  fixedOverhead?: number;
  /** 每条 chat 消息的额外 overhead（LLM Chat 用） */
  perMessageOverhead?: number;
  /** 每个工具/函数 schema 的额外 overhead */
  perToolOverhead?: number;
  /** 上下文截断时保留的安全比例（0-1） */
  reserveRatio?: number;
}

export interface TokenizerProfile {
  id: string;
  name: string;
  version: string;
  description?: string;
  /** 用于 modelId → profile 自动匹配的正则模式 */
  modelPatterns: string[];
  source: TokenizerSource;
  confidence: TokenizerConfidence;
  calibration?: TokenizerCalibration;
  license?: string;
  homepage?: string;
  tags?: string[];
}

export interface TokenizerRule {
  id: string;
  /** 匹配模型 ID 的正则 */
  pattern: string;
  profileId: string;
  priority?: number;
  enabled?: boolean;
}
```

### 6.3 `TokenCalculationResult` 扩展（additive）

```ts
export interface TokenCalculationResult {
  // ===== 现有字段保持不变 =====
  count: number; // 最终结果（已应用 calibration）
  textTokenCount?: number;
  mediaTokenCount?: number;
  imageTokenCount?: number;
  videoTokenCount?: number;
  audioTokenCount?: number;
  isEstimated: boolean;
  tokenizerName: string;

  // ===== 新增字段（v2） =====
  /** 未经 calibration 的原始 token 数（便于调试 / 展示） */
  rawCount?: number;
  /** 命中的 profile ID */
  tokenizerProfileId?: string;
  /** 置信度 */
  tokenizerConfidence?: TokenizerConfidence;
  /** 实际应用的 calibration（如果有） */
  appliedCalibration?: TokenizerCalibration;
}
```

> 关键约定：`count` 始终是「最终值」。如果配置了 multiplier=1.2，则
> `count = Math.round(rawCount * 1.2 + overhead)`。LLM Chat 等下游模块**只看
> `count`**，无需理解 calibration，行为兼容。

## 7. 解析优先级

引擎在为一个 modelId 寻找 profile 时按以下顺序：

1. 用户在「匹配规则」Tab 配置的显式 override；
2. 用户已安装 / 已下载的 profile（按 `modelPatterns` 匹配）；
3. 内置 profile（按 `modelPatterns` 匹配）；
4. 现有 [`tokenizerMappings`](../../core/tokenCalculatorEngine.ts:160) 兜底正则；
5. 字符级估算器（[`estimateTokens`](../../core/tokenCalculatorEngine.ts:280)）。

匹配后再叠加该 profile 的 calibration。

## 9. Worker 通信协议

### 9.1 消息类型

```ts
// 主线程 → Worker
type WorkerInbound =
  | { type: "init"; profiles: TokenizerProfile[]; rules: TokenizerRule[] }
  | { type: "profileData"; profileId: string; tokenizerJSON: string; tokenizerConfigJSON?: string }
  | { id: number; method: "calculateTokens" | ...; params: any };

// Worker → 主线程
type WorkerOutbound =
  | { type: "ready" }
  | { type: "initialized" }
  | { type: "needProfileData"; profileId: string }
  | { id: number; type: "response"; result: any }
  | { id: number; type: "error"; error: string };
```

### 9.2 启动流程

```
Proxy 创建 Worker
  ↓
Worker onmessage 准备就绪 → postMessage({ type: "ready" })
  ↓
Proxy 收到 ready → 从 store 拉快照 → postMessage({ type: "init", profiles, rules })
  ↓
Worker 重建内部 registry → postMessage({ type: "initialized" })
  ↓
Proxy flush 等待队列里的 calculate* 请求
```

### 9.3 重启流程（安装 / 卸载 / 改规则触发）

```
proxy.restartWorker():
  1. isRestarting = true，新请求进队列
  2. 等待 pending（超时 2s 强制 terminate）
  3. terminate 老 worker
  4. new Worker() → 走 9.2 启动流程
  5. initialized 后 flush 队列、isRestarting = false
```

### 9.4 按需加载 tokenizer.json

```
Worker 在 calculateTokens 时发现 profile 未加载
  → postMessage({ type: "needProfileData", profileId })
  → 当前请求挂起（保留 callback）

主线程 onmessage 收到 needProfileData
  → tokenizerAssetService.readProfileFiles(profileId)
  → postMessage({ type: "profileData", profileId, tokenizerJSON, ... })

Worker 收到 profileData
  → TokenizerLoader.fromPreTrained({ tokenizerJSON, tokenizerConfigJSON })
  → 缓存
  → 重新触发挂起的 calculateTokens
```

为减少传输开销，`tokenizerJSON` 可考虑用 `TextEncoder` 转 ArrayBuffer + transfer。

## 10. 下载源（Phase 5）

MVP 直接固定一个 GitHub Release Asset URL（具体仓库地址在 Phase 5 启动时确认）：

```
https://github.com/<owner>/aio-hub-tokenizer-index/releases/latest/download/index.json
```

`index.json` 结构：

```json
{
  "schemaVersion": 1,
  "updatedAt": "2026-05-30",
  "tokenizers": [
    {
      "id": "remote.gptoss-o200k-harmony",
      "name": "GPT-OSS o200k_harmony",
      "version": "1",
      "modelPatterns": ["^gpt-oss", "^openai/gpt-oss"],
      "source": {
        "type": "remote",
        "format": "hf-tokenizer-json",
        "tokenizerJsonUrl": "https://example.com/gptoss/tokenizer.json",
        "tokenizerConfigUrl": "https://example.com/gptoss/tokenizer_config.json",
        "tokenizerJsonSha256": "..."
      },
      "confidence": "exact",
      "license": "Apache-2.0"
    }
  ]
}
```

下载策略：

- 下载文件先落 `AppData/<app>/tokenizer-assets/<profileId>/tokenizer.json`；
- 索引提供 sha256 时必须校验；
- 离线场景下展示已缓存的 index，不强制重新拉取；
- 远端 URL 不在 [`index.json`](src/tools/token-calculator/docs/Plan/tokenizer-asset-registry-v2.md:1) 白名单内的，
  在 UI 上需要二次确认（避免随手贴恶意 URL）。

## 11. 导入流程

### 11.1 生态调研结论（2026-05-30）

本地导入不能只按 `@lenml/tokenizers` 当前内置包的形态设计。`@lenml/tokenizers`
的 `TokenizerLoader.fromPreTrained()` 直接要求两份对象：`tokenizerJSON` 和
`tokenizerConfig`；现有内置包也都是 `models/tokenizer.json` +
`models/tokenizer_config.json`。但用户实际从 Hugging Face、OpenAI/tiktoken、
SentencePiece、Mistral 或本地 GGUF 模型拿到的资产形态并不一致。

调研来源：

- Hugging Face Transformers tokenizer 文档与源码
  ([tokenizer API](https://huggingface.co/docs/tokenizers/en/api/tokenizer),
  [pipeline](https://huggingface.co/docs/tokenizers/python/latest/pipeline.html))：
  `save_pretrained()` 目录可能包含
  `tokenizer.json`、`tokenizer_config.json`、`special_tokens_map.json`、
  `added_tokens.json`，慢分词器还会包含 `vocab.txt`、`vocab.json` / `merges.txt`、
  `tokenizer.model` 等模型专用文件；
- Hugging Face Tokenizers：`tokenizer.json` 是 fast tokenizer 的统一序列化结果，
  里面包含 normalizer、pre_tokenizer、model、post_processor、decoder 和
  added_tokens；
- OpenAI [tiktoken](https://github.com/openai/tiktoken)：常见资产是 `.tiktoken` 文本文件，
  本质是 base64 token bytes +
  rank 的 BPE rank 表，还需要代码侧知道 regex pattern、special tokens 和 explicit
  n_vocab；
- Google [SentencePiece](https://github.com/google/sentencepiece)：训练产物通常是二进制
  `.model` 和文本 `.vocab`，模型类型可能是
  unigram、bpe、char 或 word；`.model` 不是 JSON；
- Mistral/tekken
  ([mistral-common tokenizer 文档](https://mistralai.github.io/mistral-common/usage/tokenizers/))：
  部分模型发布 `tekken.json` 或类似 tiktoken-compatible 资产，不等同于 HF
  `tokenizer.json`；
- [GGUF 规范](https://github.com/ggml-org/ggml/blob/master/docs/gguf.md)：tokenizer 信息可能
  内嵌在模型文件 metadata 中，例如 tokens、token_type、merges、bos/eos/pad id、
  chat_template 等 key，而不是独立 tokenizer 配置文件。

因此导入流程需要先做**资产探测**，再决定是直接加载、转换后加载、还是只登记为
`estimated/close` 并提示当前版本不可精确分词。

### 11.2 支持的导入入口

- 单个 `tokenizer.json`；
- `tokenizer.json` + `tokenizer_config.json`；
- Hugging Face tokenizer 目录（自动识别 fast / slow 相关文件）；
- 单个 `.tiktoken`；
- SentencePiece `.model`（可附带 `.vocab`）；
- GPT-2 / RoBERTa 类 BPE：`vocab.json` + `merges.txt`；
- BERT / WordPiece 类：`vocab.txt`；
- Mistral/tekken 类 JSON；
- GGUF 文件（只读取 metadata，不加载整模型；初版可只探测并提示后续支持）；
- 一对远端 URL（高级模式，需要确认）。

### 11.3 资产形态矩阵

| 形态                         | 常见文件                                                                                         | 能否直接走当前 `@lenml/tokenizers` | Phase 3 策略                                                                                                        |
| ---------------------------- | ------------------------------------------------------------------------------------------------ | ---------------------------------- | ------------------------------------------------------------------------------------------------------------------- |
| HF fast tokenizer            | `tokenizer.json` + `tokenizer_config.json`，可选 `special_tokens_map.json` / `added_tokens.json` | 可以，最贴近当前内置包             | 首期精确支持；缺 `tokenizer_config.json` 时尝试生成最小 config，但降级为 `close`，除非样本测试通过且特殊 token 完整 |
| HF 目录混合形态              | 上述文件 + `config.json` / `generation_config.json` / `chat_template.jinja`                      | 只消费 tokenizer 相关文件          | 目录扫描时忽略非 tokenizer 文件；`chat_template` 只存 manifest，不参与纯文本 token 计数                             |
| GPT-2 / RoBERTa BPE 慢分词器 | `vocab.json` + `merges.txt` + 可选 `tokenizer_config.json`                                       | 不能直接加载                       | 后续做转换器：生成 HF fast-style `tokenizer.json`；首期可探测并提示“需要转换支持”                                   |
| BERT / WordPiece 慢分词器    | `vocab.txt` + 可选 `tokenizer_config.json`                                                       | 不能直接加载                       | 后续做转换器：生成 WordPiece `tokenizer.json`；首期只探测                                                           |
| SentencePiece                | `tokenizer.model` / `spiece.model` / `*.model` + 可选 `*.vocab`                                  | 不能直接加载二进制 `.model`        | 后续引入 WASM/JS SentencePiece 或转换器；首期只登记为不可精确导入                                                   |
| tiktoken                     | `*.tiktoken`，有时还需要 encoding 名称、pat_str、special_tokens                                  | 不能直接加载                       | 后续实现 tiktoken adapter；仅 `.tiktoken` 不足以唯一恢复完整 encoding，UI 需要求用户选择/补全 encoding 元数据       |
| Mistral tekken               | `tekken.json` / `tokenizer.model.v3` / `tokenizer.json`（不同发布源不一致）                      | 多数不能直接加载                   | 探测为 `tekken-json`，后续接 adapter；若发布源同时给 HF `tokenizer.json`，优先使用 HF fast 路径                     |
| GGUF 内嵌 tokenizer          | 单个 `*.gguf`，metadata 中带 tokenizer key                                                       | 不能直接加载                       | 首期最多读取 header/metadata 做识别；不复制整模型进 tokenizer-assets，避免巨大文件                                  |

### 11.4 规范化导入模型

目录或文件导入后，先生成内部 manifest，再决定如何加载：

```ts
export interface TokenizerImportScanResult {
  format: TokenizerAssetFormat;
  files: Record<string, string>;
  warnings: string[];
  loadability: "direct" | "convertible" | "unsupported";
  suggestedConfidence: TokenizerConfidence;
  detectedTokenizerClass?: string;
  detectedModelType?: string;
  detectedSpecialTokens?: string[];
}
```

`tokenizerAssetService` 只把导入目录中的**必要 tokenizer 资产**复制到
`AppData/<app>/tokenizer-assets/<profileId>/`，不整目录复制。对 GGUF 这种大模型文件，
默认只记录来源路径和提取出的 metadata 摘要，不复制原文件。

### 11.5 首期支持边界

Phase 3 首期建议只把以下形态作为“可启用”的精确导入：

- `tokenizer.json` + `tokenizer_config.json`；
- Hugging Face fast tokenizer 目录中能找到上述两份文件；
- 单个 `tokenizer.json`，但必须能补出最小 `tokenizer_config` 并通过样本计数测试；
- 远端 URL 高级模式也仅限上述 JSON 形态。

其它形态先做到**识别清楚 + 给出下一步提示**，不要假装导入成功：

- `.tiktoken`：提示需要 tiktoken adapter / encoding 元数据；
- `.model`：提示需要 SentencePiece adapter；
- `vocab.json + merges.txt` / `vocab.txt`：提示需要慢分词器转换；
- `tekken.json`：提示需要 tekken adapter；
- `*.gguf`：提示当前只识别 metadata，不做精确 token 计数。

### 11.6 导入对话框收集

- 显示名 / Profile ID（自动生成，可改，强制 `user.` 前缀）；
- `modelPatterns` 正则；
- `confidence`；
- 可选的 `calibration`。

导入完成后自动跑一次「样本计数测试」（用预设短文本调用 Worker 验证 profile 可用），通过后再启用。
样本测试至少覆盖 ASCII、中文、emoji、换行、常见特殊 token；如果用户导入了 config 中声明的
special token，还要验证这些 token 不会被拆坏。

## 12. 校准流程

某些供应商没有公开本地分词器（典型如新 Claude），通过 calibration 提供保守估算：

- `multiplier`：例如 1.2，弥补本地估算偏低；
- `fixedOverhead`：每次调用的固定 token 开销；
- `perMessageOverhead`：每条 chat message 的固定开销；
- `perToolOverhead`：每个工具 schema 的固定开销；
- `reserveRatio`：上下文截断时保留的安全比例。

UI 展示规则：

- 只要应用过 calibration，结果就标注 `估算 · claude · ×1.20`；
- 同时显示 `rawCount` 和 `count`，便于用户判断；
- LLM Chat 内消息体的 token 显示，沿用 `count`，不需要感知 calibration。

## 13. 安全 / 校验

- 下载文件只接受 JSON；
- 不执行远端代码；
- 用 **Zod**（项目已有先例）校验 profile schema；
- 解析前先做文件大小限制（默认 50 MB）；
- 提供 sha256 时必须校验；
- 禁止 `fetch(dataUrl)`（Tauri CSP 限制，详见全局开发规范）；
- 安装记录里保留来源 URL、license、安装时间，UI 可见。

`gpt4o` / `gpt4` / `claude` / `gemini` / `llama3_2` / `deepseek_v3` / `qwen3`

### ✅ Phase 4.5：内置 Profile 用户覆盖持久化 — 已完成

> **背景**：在 Phase 4.5 之前，用户在 UI 上禁用一个内置 profile（比如禁用近似的
> `claude` 让它走 estimator）只能在当前会话生效，应用重启后会被代码层的
> `enabled: true` 重置。Phase 6 校准也面临同样问题——内置 profile 的 calibration
> 也需要落到磁盘。Phase 4.5 把两者合并到一套"用户覆盖"机制中。

实现要点：

- 类型层新增 [`BuiltinProfileOverride`](../../types/tokenizer-profile.ts:152)：
  ```ts
  interface BuiltinProfileOverride {
    enabled?: boolean;
    calibration?: TokenizerCalibration;
  }
  ```
- `profiles.json` 持久化结构新增 `builtinOverrides: Record<string, BuiltinProfileOverride>`，
  键为内置 profile ID。
- [`tokenizerRegistryStore`](../../stores/tokenizerRegistryStore.ts:1) 内：
  - `builtinDefaults` = 启动时从 `BUILTIN_TOKENIZERS` 重建的不变量;
  - `builtinProfiles` = 应用覆盖后的生效快照（UI / engine / Worker 看到的内容）;
  - `applyBuiltinOverrides()` = 从 `builtinDefaults` + `builtinOverrides` 重建 `builtinProfiles`;
  - 启动 `load()` 时读取并应用覆盖；
  - 所有内置 profile 的写操作都走 `updateBuiltinOverride() + persistAndRestart()`，
    与用户 profile 的写路径统一。
- 仅记录"差异"（覆盖项与代码默认值不同的字段），用户把字段调回默认值时自动清理键，
  防止累积无用数据。
- 升级 / 卸载内置 profile 时，`builtinOverrides` 中残留的 ID 是无害的（找不到 default
  时直接跳过）。
- API 暴露：[`setProfileEnabled`](../../stores/tokenizerRegistryStore.ts:346) / [`setProfileCalibration`](../../stores/tokenizerRegistryStore.ts:376) / [`resetBuiltinOverride`](../../stores/tokenizerRegistryStore.ts:408) / [`hasBuiltinOverride`](../../stores/tokenizerRegistryStore.ts:419)。
- [`TokenizerLibraryPanel`](../../components/library/TokenizerLibraryPanel.vue:1) 在被覆盖的内置 profile 上显示「重置」按钮，
  一键清除该 ID 的全部覆盖。

`profiles.json` 的 `builtinOverrides` 字段中，启动时合并到内置 profile 快照上

## 18. 验收标准

- 用户无需重新打包应用即可安装 / 导入新 profile；
- LLM Chat 在所有现有路径下 token 计数行为与旧版一致；
- Token 计算结果在 UI 上能清晰看到来源（profile / 内置 / 估算）和置信度；

- 注册表变更操作（安装 / 卸载 / 改规则）触发的 Worker 重启对 UI 无感（loading 提示 < 500ms）。
