---

## 2. 三种模式的本质差异

| 维度         | Quick / Render Mode               | Interactive Mode                         |
| :----------- | :-------------------------------- | :--------------------------------------- |
| **用户**     | Agent / 程序化调用 / 人类一键操作 | 仅人类                                   |
| **交互模型** | 管道式：输入 URL → 吐出内容       | 画布式：在真实网页上探索、标记、配方化   |
| **UI 需求**  | 结果预览面板 + 质量指标           | 全屏浏览器 + 覆盖层工具 + 侧边属性面板   |
| **核心价值** | 快速获取干净内容                  | 为复杂网站创建可复用的提取规则           |
| **生命周期** | 一次性：请求 → 响应 → 结束        | 持续性：加载 → 探索 → 标记 → 调试 → 保存 |

当前代码把它们硬塞进 `DistilleryWorkbench.vue` 的 `v-if/v-else`，导致交互模式的 UI 被压缩在一个小面板里，既没有空间展示完整网页，也没有空间放操作工具。

## 5. 交互模式应独立设计的理由

### 5.1 产品形态不同

Quick/Render 是"自动售货机"——投币取货。Interactive 是"实验室"——需要工作台、工具、试管架。把实验室塞进自动售货机的柜子里，两边都不好用。

### 5.2 UI 空间需求不同

交互模式需要：

- **左侧大面积**：完整网页渲染 + 覆盖层工具（高亮、拾取、删除）
- **右侧工具面板**：配方编辑器 + 实时预览 + API 嗅探结果
- **底部状态栏**：选中元素信息、Cookie 状态、加载进度

这些在当前 `DistilleryWorkbench.vue` 的 `v-if` 分支里根本放不下。

### 5.3 状态管理不同

Quick/Render 是无状态的——请求完就扔。Interactive 需要维护：

- 当前加载的 URL 和页面状态
- 拾取器模式（包含/排除/动作）
- 正在编辑的配方草稿
- 实时预览内容（随规则变动更新）
- 嗅探到的 API 列表

## 6. 交互模式落地需要解决的核心问题

以下不是完整设计方案，而是指出"缺什么"：

### 6.1 用户怎么进入交互模式？

当前：在工作台里按一个按钮切换 `isInteractiveMode`
问题：切换后 UI 突变，用户不知道发生了什么

### 6.2 用户进来后该做什么？

当前：看到一个 Iframe + 一个空的 RecipeEditor
问题：没有引导，用户不知道第一步是什么

### 6.3 用户怎么标记元素？

当前：selector-picker.js 存在但通信断裂
需要：修复通信 + 设计"拾取模式"的交互流程（进入拾取→悬停高亮→点击确认→回填到规则列表）

### 6.4 用户怎么看到效果？

当前：只有保存配方后用 Quick/Render 模式才能看到效果
需要：实时预览——每改一条规则，右侧立即显示提取结果的变化

### 6.5 用户怎么录制动作序列？

当前：手动在 RecipeEditor 里填表
需要：至少支持"录制模式"——记录用户的点击和滚动，自动转为 ActionStep

### 6.6 配方怎么从交互模式流向自动模式？

当前：recipe-store 的 CRUD 是通的，但从交互模式创建配方的入口很别扭
需要：交互模式的"保存"应该自然地生成一个完整的 SiteRecipe 并持久化
